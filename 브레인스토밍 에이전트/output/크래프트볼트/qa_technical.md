### 3-3. 다운로드 함수 구현

**`downloadAll()` 함수:**
- slides 전체 querySelectorAll → html2canvas(scale:2) → JSZip → blob URL download
- 출력 캔버스: 3840×2160 (2배 배율)
- `captureSlide()`: letterbox 방식으로 슬라이드를 4K 캔버스에 맞춤 배치
- `saveCanvas()`: PNG 다운로드 정상 구현

**`toggleIndividual()` 함수:**
- 각 슬라이드 하단에 저장 버튼 동적 생성/제거
- `individualMode` 토글 변수로 상태 관리

**지적 사항:**
- `slideNames` 배열이 40개 선언 → 실제 슬라이드도 40개(slide-01~slide-41, slide-31 누락으로 실질 40개). 1:1 매칭 ✅
- `zipFilename`에 한글 포함 → 대부분의 브라우저에서 인코딩 정상 처리되나, 일부 구형 브라우저/OS에서 파일명 깨짐 가능성 존재 (⚠️ 경미)
- `downloadAll()` 비동기 처리가 순차(for + await)로 구현 → 40슬라이드 순차 캡처 시 처리 시간이 길어질 수 있음 (⚠️ UX 고려사항, 버그 아님)

### 3-4. @media print 스타일

```css
@media print {
  .top-nav, .slide-dl-btn { display:none !important; }
  body { padding:0; background:white; }
  .slide-container { box-shadow:none; page-break-after:always; }
}
```

| 검수 항목 | 판정 |
|----------|------|
| 네비게이션 숨김 | ✅ `.top-nav` display:none |
| 다운로드 버튼 숨김 | ✅ `.slide-dl-btn` display:none |
| 슬라이드 페이지 분리 | ✅ `page-break-after:always` |
| body 배경 흰색 | ✅ |

**print 스타일: 정상.** 단, 1920px 고정 너비 슬라이드를 A4 용지에 인쇄하면 축소 비율이 발생하므로 인쇄 목적으로는 별도 최적화 필요 (설계 한계, 버그 아님).

## 4. 슬라이드 구조 불일치 — 중요 이슈

### 4-1. 슬라이드 ID vs. 스크립트 번호 불일치

스크립트의 슬라이드 인덱스와 HTML `id` 간에 번호 체계가 어긋남:

| 스크립트 번호 | 슬라이드 제목 | HTML id | HTML 표시 번호 |
|-------------|------------|---------|-------------|
| 5-32 (KPI 대시보드) | KPI 대시보드 | `slide-32` | `31 / 38` |
| 5-33 (매출 시나리오) | 매출 시나리오 분석 | `slide-33` | `32 / 38` |
| ... | ... | `slide-34`~`slide-37` | `33~36 / 38` |
| 6-38 (3가지 이유) | 왜 크래프트볼트인가 | `slide-38` | `37 / 38` |
| 6-39 (투자 효과) | 투자 효과 요약 | `slide-39` | `38 / 38` |
| 6-40 (킥오프) | 실행 일정 확정 | `slide-40` | `39 / 38` |
| 6-41 (클로징) | 클로징 커버 | `slide-41` | `40 / 38` |

**`slide-31`이 HTML에 존재하지 않음.** 전체 슬라이드 수는 40개이나 id 시퀀스에 31이 빠져 있음 (`slide-30` 다음이 `slide-32`). 이로 인해:
- `slide-32`(KPI 대시보드)의 화면 표시 번호가 `31 / 38`로 표기됨
- `slide-40`(킥오프)의 화면 표시 번호가 `39 / 38`로 표기됨
- `slide-41`(클로징)의 화면 표시 번호가 `40 / 38`로 표기됨

**결론:** 전체 슬라이드 수가 38장이 아닌 40장이 구현됨 (커버+클로징 포함 실질 40장). 스크립트 인덱스 4-30과 4-31(리스크 관리)이 혼재되어 있어 slide-30과 slide-31의 간격이 발생한 것으로 보임. **"39 / 38", "40 / 38" 표기는 사용자에게 혼란을 줄 수 있는 명시적 버그.**
